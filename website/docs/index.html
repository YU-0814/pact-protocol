PACT Protocol Specification

Version 1.0 — PACT Working Group 2026

한국어 | English

This is a summary of the PACT specification. For the full normative spec, see the GitHub repository.

Overview

PACT (Protocol for Agent Content Transfer) is an open standard for serving structured, token-efficient content to AI agents. It fills the gap between discovery standards (robots.txt, llms.txt) and interaction protocols (MCP, A2A).

Where MCP gives AI agents tools and A2A connects agents together, PACT solves the content delivery problem: how do websites serve their actual data (products, articles, listings) to AI agents in a format that is compact, typed, and actionable?

Design Principles

  • Token efficiency — minimize tokens consumed by AI models (57x reduction vs HTML)
  • Progressive adoption — start with a single JSON file, grow incrementally
  • Schema-driven — typed data with versioned schemas for reliable parsing
  • Publisher control — site owners define terms, pricing, and access rules
  • Cross-platform — works on web, mobile, IoT — anywhere HTTP speaks

Architecture

PACT uses a 5-layer stack, where each layer builds on the previous one. Implementations can adopt layers incrementally.

Layer Name Purpose Key File / Field
0 Discovery Agents find your PACT endpoints /.well-known/pact.json
1 Schema Data types are identified and versioned pact:domain/type@version
2 Data Structured JSON content responses $pact, $s, items
3 Media Image and video references URL pointers in data fields
4 Actions Transactional operations $actions array

Layer 0: Discovery

A JSON file at /.well-known/pact.json tells agents what content your site offers, what schemas it uses, and where to request data.

Layer 1: Schema

Every PACT response identifies its data type using a schema URI in the format pact:domain/type@version. Agents use this to parse and understand the structure without guessing.

Layer 2: Data

The actual content — structured JSON with typed fields. Supports both object mode (items array) and tabular mode (cols + rows) for maximum token efficiency.

Layer 3: Media

Media assets (images, videos, documents) are referenced by URL rather than embedded, keeping response payloads small and letting agents selectively fetch what they need.

Layer 4: Actions

The $actions array lets sites expose transactional operations — purchase, reserve, subscribe, etc. — as typed endpoints with defined parameters.

Discovery Protocol

Every PACT-enabled site hosts a discovery document at /.well-known/pact.json. This is the entry point for AI agents.

Required Fields

FieldTypeDescription
pactstringProtocol version (e.g., "1.0")
sitestringDomain name (e.g., "example.com")
schemasstring[]List of supported schema URIs
endpointsRecordMap of schema IDs to endpoint definitions (list, item, search)

Optional Fields

FieldTypeDescription
descriptionstringHuman-readable site description
platformsRecordPlatform-specific config (web, ios, android)
rate_limitobjectRate limiting: rpm, burst
authobjectAuthentication: type, register
licenseobjectUsage terms: ai_input, ai_train, attribution
conformancestringSelf-declared level: L1, L2, L3, or L4

Example

{
  "pact": "1.0",
  "site": "store.example.com",
  "description": "Online electronics store",
  "schemas": ["pact:commerce/product@1"],
  "endpoints": {
    "pact:commerce/product@1": {
      "list": "/pact/products",
      "item": "/pact/products/{id}",
      "search": "/pact/products?q={query}"
    }
  },
  "rate_limit": { "rpm": 60, "burst": 10 },
  "license": { "ai_input": true, "ai_train": false, "attribution": true }
}

Schema System

PACT schemas follow the URI format pact:domain/type@version, providing a universal naming convention for data types.

Schema URI Format

pact:domain/type@version

Examples:
  pact:commerce/product@1
  pact:content/article@1
  pact:local/restaurant@1
  pact:media/video@1

Standard Domains

DomainCoversExample Types
commerceE-commerce, productsproduct, price-comparison, store
contentArticles, blogs, docsarticle, blog-post, documentation
localLocal businessesrestaurant, hotel, service
mediaMedia contentvideo, podcast, image-gallery

Data Layer

PACT responses use a minimal envelope format designed for token efficiency. Two modes are supported:

Object Mode

Standard JSON with an items array. Compressed keys for token efficiency.

{
  "$pact": "1.0",
  "$s": "pact:commerce/product@1",
  "$t": 1710590400,
  "$ttl": 300,
  "items": [
    { "id": "p1", "n": "Widget", "p": 9.99, "stk": true },
    { "id": "p2", "n": "Gadget", "p": 19.99, "stk": true }
  ],
  "total": 2,
  "page": { "offset": 0, "limit": 20 }
}

Tabular Mode

Columns + rows for maximum compression. Best for large, uniform datasets.

{
  "$pact": "1.0",
  "$s": "pact:commerce/product@1",
  "$t": 1710590400,
  "$layout": "table",
  "cols": ["id", "n", "p", "stk"],
  "rows": [
    ["p1", "Widget", 9.99, true],
    ["p2", "Gadget", 19.99, true]
  ],
  "total": 2
}

System Fields

FieldRequiredDescription
$pactYesProtocol version
$sYesSchema URI (pact:domain/type@version)
$tYesUnix timestamp of response generation
$ttlNoCache TTL in seconds
$layoutNoSet to "table" for columnar format
totalNoTotal count of matching items
pageNoPagination: offset, limit, next

Actions

The $actions array lets sites expose transactional endpoints directly within PACT responses. This allows AI agents to perform operations like purchasing, reserving, or subscribing on behalf of users.

Action Structure

Actions use verb (HTTP method) and name (action identifier) per the spec:

{
  "actions": [
    {
      "verb": "POST",
      "name": "purchase",
      "description": "Add to cart",
      "url": "/pact/cart/add",
      "body": { "product_id": "string", "quantity": "number" },
      "auth_required": true
    },
    {
      "verb": "POST",
      "name": "reserve",
      "description": "Make a reservation",
      "url": "/pact/reservations",
      "body": { "date": "string", "time": "string", "party_size": "number" }
    }
  ]
}

Standard Action Types

  • purchase — buy a product or service
  • reserve — make a reservation
  • subscribe — start a subscription
  • contact — send a message or inquiry
  • bookmark — save or favorite an item

Conformance Levels

PACT defines four conformance levels to support progressive adoption. Each level builds on the previous.

LevelNameRequirementsEffort
L1 Discoverable Valid pact.json at /.well-known/pact.json ~5 minutes
L2 Typed L1 + schema identifiers, structured responses with $pact and $s ~30 minutes
L3 Efficient L2 + tabular mode, pagination ($meta), compression Hours
L4 Trusted L3 + authentication, $actions, licensing, agent verification Days

Cross-Platform Support

PACT responses can declare target platforms using the $platforms field. This enables agents to deliver platform-specific content when available.

{
  "$pact": "1.0",
  "$s": "pact:local/restaurant@1",
  "$platforms": ["web", "ios", "android"],
  "items": [...]
}

Supported Platforms

  • web — browser-based web applications
  • ios — Apple iOS applications
  • android — Android applications
  • api — headless API consumers
  • iot — IoT devices and embedded systems

Security Considerations

PACT includes provisions for secure content delivery at L4 conformance:

  • Authentication — API key, OAuth 2.0, or custom auth for protected endpoints
  • Agent verification — verify the identity and permissions of requesting agents
  • Rate limiting — control request frequency per agent or key
  • Content licensing — define usage terms and attribution requirements in discovery
  • HTTPS required — all PACT endpoints should use TLS encryption
